> **STATUS: DONE (2026-06-13).** Both tasks complete, hardened, accelerated, and finalized.
>
> **Results (GTX 1650, sm_75):**
> - **Throughput: 306.2 M keys/s** (median of 3 × 8 s `--bench` runs), vs **25.4 M/s** for the 12-thread CPU incremental baseline → **12.0× whole-CPU** (clears the ≥10× Phase-3 gate).
> - **Device field:** native **8×32-bit limbs** (`device_field32.cuh`, `ONION_CUDA_FIELD32=ON`) replaced the `__int128` 5×51 field (kept as the cross-checked reference). Search kernel: **178 → 96 registers/thread, 0 spills** (`ptxas -v`); that occupancy gain drove ~275 → ~306 M/s.
> - **Correctness:** CUDA suite **56/56 green**, CPU debug suite **53/53 green**. The device chain matches libsodium `base_noclamp(a0+8i)` bit-for-bit; every key passes the Verifier firewall + Python oracle.
> - **Leak/safety:** `compute-sanitizer --tool memcheck --leak-check full` → **0 bytes leaked / 0 errors** on both the test binary and a real generate run (per-epoch `cudaMalloc`/free loop is RAII-clean).
>
> Original plan below, for the record.

> Gated by **bit-exact cross-validation against libsodium** on the host, exactly like the CPU phases. A fast-but-wrong GPU generator is worthless.

**Hardware/toolchain (proven working):** NVIDIA GTX 1650 (Turing, sm_75, 14 SMs, 4 GB). `nvcc` 13.3 at `/opt/cuda/bin/nvcc`, host compiler **g++-15** (`-ccbin g++-15`; `NVCC_CCBIN` is preset system-wide). `unsigned __int128` **compiles and runs correctly in device code** on sm_75 — so the device field reuses the validated 5×51 donna64 algorithm (correctness-first; a 32-bit-limb rewrite is a later perf option).

**Goal:** A `CudaEngine` (behind `IEngine`) that searches on the GPU: tens of thousands of threads, each walking its own incremental chain `A += 8B`, matching leading y-bytes on-device, reporting hits to the host for the verifier firewall. Target: a large multiple of the ~25 M/s CPU (a GTX 1650 realistically lands a few hundred M/s for this IMAD-heavy workload — measure, don't promise).

**Toolchain firewall (design §6):** the CUDA TUs are an OPTIONAL CMake target (`option(ONION_CUDA ...)`, default OFF; ON when nvcc is found). They compile with nvcc+g++-15; the rest of the project stays on GCC 16. The boundary header is plain C++ (structs + byte spans, no `std::expected`/`std::print`); host C++23 never includes CUDA headers. Graceful degrade to CPU-only when nvcc/GPU absent.

### Task 1 — CMake CUDA enablement + device crypto core + libsodium cross-validation

**Files:** `CMakeLists.txt` (+ `ONION_CUDA` option), `src/engine/cuda/CMakeLists.txt`, `src/engine/cuda/device_field.cuh` (device fe/ge), `src/engine/cuda/xval_kernel.cu` (+ a tiny host harness), `tests/test_cuda_xval.cpp` (or a standalone runner registered with ctest).

1. **CMake:** add `option(ONION_CUDA "Build the CUDA backend" OFF)`. When ON: `enable_language(CUDA)`, set `CMAKE_CUDA_ARCHITECTURES 75`, `CMAKE_CUDA_HOST_COMPILER` to `g++-15` (or rely on `NVCC_CCBIN`), `CMAKE_CUDA_STANDARD 20`. Add `src/engine/cuda` subdir building a `onion_engine_cuda` target from the `.cu` files. Add a CMake preset `cuda` (Release + ONION_CUDA=ON). Guard everything so the default presets still build CPU-only with nvcc absent.
2. **Device field/point (`device_field.cuh`):** `__device__` 5×51 field (`fed_mul/sq/add/sub/invert/frombytes/tobytes`) — port the validated `fe25519.cpp` algorithm verbatim (it uses `unsigned __int128`, which works on device), and the `ged_madd` mixed addition + a device `scalarmult_base` (or pass A0 in from the host). Keep it header-only `__device__` inline.
3. **Cross-validation kernel + test (THE GATE):** a kernel that, given a base scalar `a0` and `N`, computes the incremental chain `A0, A0+8B, …` on ONE thread (or a few), writing each point's y-bytes back. The host test (`test_cuda_xval`, links libsodium) checks, for several random clamped `a0` and `i` in `0..N`: device y-bytes[i] (leading 31) == `crypto_scalarmult_ed25519_base_noclamp(a0+8i)`. **Never weaken.** This proves the device crypto matches libsodium bit-for-bit, just like the CPU keystone.
4. Register the test with ctest **only when ONION_CUDA is ON** (so CPU-only CI is unaffected). Build+run: `cmake --preset cuda && cmake --build --preset cuda && ctest --preset cuda -R cuda`.

**Verify:** the xval kernel matches libsodium on all sampled points; `compute-sanitizer ./<runner>` clean (memcheck) if available.

### Task 2 — Search kernel + `CudaEngine` + CLI integration + measure

**Files:** `src/engine/cuda/search_kernel.cu`, `src/engine/cuda/cuda_engine.{hpp,cpp}` (host wrapper, `class CudaEngine : public IEngine` — the .hpp is plain C++ included by the CLI; the .cpp/.cu owns all CUDA calls), `src/cli/main.cpp` (`--engine cuda`).

- **Epoch setup:** host draws a random clamped `a0`; each thread `t` gets start scalar `a0 + 8·t·K` and computes its start point (device `scalarmult_base`, or host computes per-thread starts and uploads). K = candidates/thread/launch.
- **Search kernel:** each thread walks `A += 8B` for K steps (per-thread short Montgomery inversion chain, M≈16–32, to amortize the inversion — or a first version with per-step inversion for simplicity, then optimize), serializes leading y-bytes, masked-compares against the compiled patterns (uploaded to `__constant__`). On a hit: `atomicAdd` a global counter + write `(thread, iter)` to a small slot buffer.
- **Host loop:** repeated grid launches (~50–200 ms each); read back the hit counter+slots; for each hit reconstruct the scalar (`a0 + 8·(t·K + iter)`), build a `MatchCandidate` (fresh prf_prefix), push to the SAME `ResultQueue` → the **verifier firewall re-derives via libsodium** before any key is written. Update `StatsBoard` with K·threads·launches.
- **CLI:** `--engine cuda` selects it (error cleanly if built without ONION_CUDA). Keep CPU engines default.
- **Measure:** `--bench` on the GPU; report M keys/s and the multiple over the CPU. Generate a real prefix on GPU and validate with `tools/verify_onion.py`.
- **Tune (after correctness):** registers/`__launch_bounds__`, M (chain length), threads/blocks, `__constant__` for `8B` + patterns; Nsight Compute occupancy. Document measured throughput in README.

**Risk/notes:** register pressure (extended point = 4 fe × 5 limbs) → low occupancy is expected on Turing; that's fine if IMAD-bound. `__int128` device math is emulated and not optimal — if throughput disappoints, the perf path is a 32-bit-limb (`mul.lo/hi`) device field, but only after the `__int128` version is correct and measured. Every hit still passes the host verifier, so even a kernel bug cannot emit a bad key.
